API Architecture - Current Project Status

Base server
- FastAPI app with CORS enabled.
- Frontend origin comes from FRONTEND_URL, defaulting to http://localhost:5173.
- Auth uses Bearer access tokens in the Authorization header plus an httpOnly refresh_token cookie.
- Google login redirects back to the frontend without exposing tokens in the URL.

Common auth rules
- Protected endpoints require Authorization: Bearer <access_token>.
- refresh_token is stored in an httpOnly cookie at path /auth.
- For browser calls that rely on refresh, use credentials: include.
- Access tokens are not returned in response bodies anymore.

Auth endpoints
- GET /
	- Health check.
	- Response: {"status": "Application is running successfully :)"}

- POST /auth/register
	- Body:
		- email: string
		- username?: string
		- full_name?: string
		- password: string
	- Response:
		- user: PublicUserSchema
		- message: string

- POST /auth/login
	- Body:
		- identifier: string
		- password: string
	- Response:
		- message: string
		- user: PublicUserSchema
	- Side effect: sets refresh_token cookie and returns access token in the Authorization response header.

- POST /auth/logout
	- Header: Authorization: Bearer <access_token>
	- Response:
		- message: string
	- Side effect: invalidates session and clears refresh cookie.

- POST /auth/refresh
	- Uses refresh_token cookie.
	- Response:
		- message: string
		- user: PublicUserSchema
	- Side effect: rotates refresh cookie and returns a new access token in the Authorization response header.

- GET /auth/me
	- Header: Authorization: Bearer <access_token>
	- Response: PublicUserSchema

- POST /auth/request-otp
	- Body:
		- email: string
	- Response:
		- email: string
		- message: string
	- Note: OTP is currently console-only, not emailed.

- POST /auth/verify-otp
	- Body:
		- email: string
		- otp_code: string
	- Response:
		- message: string
		- user: PublicUserSchema
	- Side effect: sets refresh_token cookie and returns access token in the Authorization response header.

- POST /auth/request-password-reset
	- Body:
		- email: string
	- Response:
		- email: string
		- message: string

- POST /auth/reset-password
	- Body:
		- email: string
		- reset_token: string
		- new_password: string
	- Response:
		- message: string

- GET /auth/google
- GET /auth/google/login
	- Redirects to Google OAuth consent screen.

- GET /auth/google/callback?code=...&state=...
	- Browser redirect back to frontend.
	- Query parameters:
		- status=success
		- is_new_user=true|false
	- Side effect: sets refresh_token cookie and does not expose tokens in the URL.

File ingestion endpoints
- POST /v1/upload/init
	- Header: Authorization: Bearer <access_token>
	- Body:
		- filename: string
		- filesize: integer
		- file_fingerprint: string (64 hex chars) (optional)
	- Response:
		- upload_id: string
		- filename: string
		- filesize: integer
		- chunk_size: 5242880
		- total_chunks: integer
	- Note: send the original filename including extension; the backend stores the merged file using that name when the merged file is newly created. Chunks are stored under D:/Zenteiq/Tasks/uploads/.parts and merged files are stored directly under D:/Zenteiq/Tasks/uploads.

- POST /v1/upload/chunk
	- Header: Authorization: Bearer <access_token>
	- Required header: Range
	- Multipart form-data:
		- upload_id: string
		- chunk_index: integer
		- chunk_hash: string (64 hex chars)
		- chunk_file: file
	- Response:
		- status: string
		- upload_id: string
		- chunk_index: integer
		- bytes_received: integer
		- received_chunks: integer
		- total_chunks: integer
		- chunk_size: 5242880
		- complete: boolean
	- Note: each chunk is validated and stored; finalization is performed separately by calling `/v1/upload/finalize`.

- POST /v1/upload/finalize
	- Header: Authorization: Bearer <access_token>
	- Body:
		- upload_id: string
	- Response:
		- status: "completed" | "incomplete"
		- upload_id: string
		- complete: boolean
		- queued: boolean
		- missing_indices?: list of integers
	- Note: the backend checks Redis completion first. If some chunks are still missing, it returns the missing chunk indices so the client can resend them. When all chunks are present, it queues merge/cleanup work and returns a queued ack.

- GET /v1/uploads?page=1&limit=10
	- Header: Authorization: Bearer <access_token>
	- Response:
		- page: integer
		- limit: integer
		- total_files: integer
		- files:
			- mapping_id: integer
			- filename: string
			- filesize: integer
	- Note: the list only includes uploads owned by the authenticated user.

- POST /v1/uploads/delete
	- Header: Authorization: Bearer <access_token>
	- Body:
		- mapping_id: integer
	- Response:
		- status: string
		- mapping_id: integer
		- deleted: boolean
		- message: string

Frontend needs
- Auth screens:
	- register
	- login
	- OTP verification
	- forgot password
	- reset password
	- Google sign-in callback handling

- Auth state handling:
	- store access token client-side
	- attach Authorization header for protected calls
	- call /auth/refresh when access token expires
	- keep refresh cookie support enabled in browser requests

- Upload UI:
	- file picker
	- chunk splitter with 5 MiB chunks
	- sequential upload progress indicator
	- retry/error handling for chunk failures
	- finalize button/state

- Upload history UI:
	- paginated file list
	- filename and filesize display

Important notes
- There are no download, update, or file detail endpoints yet.
- Uploads are segregated per user and users can only see their own uploaded files.
- Upload chunks must be sent sequentially.
- OTP delivery is console-only in the current implementation.
- If a different file already exists with the same final name, the backend returns a conflict instead of overwriting it.




# Frontend integration notes:
#
# Endpoints exposed by the current backend:
#   POST /v1/upload/init
#     - Request body JSON:
#         {
#           "filename": "string",
#           "filesize": integer,
#           "master_hash": "64-char hex string",
#           "relative_path": "optional relative path string"
#         }
#     - Response JSON:
#         {
#           "upload_id": "uuid string",
#           "chunk_size": 5242880,
#           "total_chunks": integer,
#           "status": "created" | "duplicate_short_circuit"
#         }
#     - Flow: frontend calls this first to create an upload session. It receives upload_id and total_chunks.
#
#   POST /v1/upload/chunk
#     - Request type: multipart/form-data
#     - Form fields:
#         upload_id: string
#         chunk_index: integer
#         chunk_hash: string (SHA-256 hex of the chunk bytes)
#         chunk_file: file (binary chunk content)
#     - Response JSON:
#         {
#           "status": "success",
#           "upload_id": "string",
#           "chunk_index": integer,
#           "bytes_received": integer,
#           "received_chunks": integer,
#           "total_chunks": integer,
#           "chunk_size": 5242880,
#           "complete": boolean
#         }
#     - Flow: frontend uploads chunks sequentially (or in parallel) and can track progress via received_chunks/total_chunks.
#
#   POST /v1/upload/finalize
#     - Request body JSON:
#         {
#           "upload_id": "string",
#           "master_hash": "64-char hex string"
#         }
#     - Response JSON:
#         {
#           "status": "success",
#           "file_id": "uuid string",
#           "folder_id": "string|null"
#         }
#     - Flow: frontend calls this once all chunks are uploaded. Backend validates chunk completion, merges the chunks, and persists the file metadata.
#
#   POST /v1/uploads/delete
#     - Request body JSON:
#         {
#           "upload_id": "string"
#         }
#     - Response JSON:
#         {
#           "status": "deleted",
#           "file_id": "string"
#         }
#     - Flow: frontend calls to delete an uploaded file by its file_id.
#
#   GET /v1/uploads
#     - Response JSON:
#         {
#           "id": "root folder id",
#           "type": "folder",
#           "name": "root",
#           "children": [ ... nested folders and files ... ]
#         }
#     - Flow: frontend calls to retrieve the user's nested upload tree. Files have type "file" and include name/size.
#
# Authentication:
#   - All endpoints require Authorization header: "Bearer <token>"
#   - The backend resolves the current user from the bearer token before processing.
#
# Frontend implementation details:
#   - Compute SHA-256 master_hash for the entire file before init.
#   - Split the file into chunks of 5,242,880 bytes (5 MB).
#   - For each chunk, compute SHA-256 chunk_hash and send with the chunk payload.
#   - Track upload_id, chunk_index, and progress from upload_chunk responses.
#   - After all chunks are uploaded, call finalize_upload to complete the file ingestion.
#   - Use GET /v1/uploads to display the virtual folder/file tree.
#
# Error handling:
#   - 404 if upload session not found
#   - 400 for invalid request data or hash mismatch
#   - 403 for authorization/user mismatch
#   - 409 if finalize is called before all chunks uploaded
#
# Important note:
#   - relative_path is user-scoped and creates nested folders in the virtual tree.
#   - physical files are stored flat under uploads/files/ by generated UUID.
